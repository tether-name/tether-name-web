# Tether.name — Agent Identity Verification

**Base URL:** `https://api.tether.name`
**Website:** [tether.name](https://tether.name)
**Protocol:** `tether-name-challenge`

## What is Tether.name?

Tether.name lets anyone — human or agent — verify that an AI agent genuinely represents who it claims to. It uses a
challenge-response protocol with RSA signatures.

## Roles

- **Challenger** — The party who wants to verify an agent's identity. Generates the challenge. Can be a human or another
  agent.
- **Prover** — The agent being verified. Signs the challenge with its private key.

**Important:** The prover must never generate its own challenge. The challenger always initiates.

## Prerequisites (for the Prover)

Your owner (the human who registered you) must have:

1. Created an account at [tether.name](https://tether.name)
2. Registered you as an agent from their dashboard
3. Given you your **agent ID** and either:
   - A **registration token** — you generate your own RSA-2048 key pair and upload your public key (recommended), OR
   - A **private key** (base64-encoded RSA PKCS8 DER format) — generated by the browser fallback flow

If you don't have these, ask your owner to set you up at [tether.name](https://tether.name).

### Activating Your Agent (Agent-Generated Key)

If your owner gave you a registration token, generate your own RSA-2048 key pair and register your public key:

1. Generate an RSA-2048 key pair (PKCS8/DER format)
2. Store the private key securely — **never share it with anyone, including your owner**
3. Upload your public key (base64-encoded X.509/SPKI):

```
POST https://api.tether.name/agents/{agentId}/register-key
Content-Type: application/json

{
  "registrationToken": "<your registration token>",
  "publicKey": "<base64-encoded public key>"
}
```

The registration token expires after 5 minutes. Once registered, you're ready to verify.

## Verification Flow

### If you are the Prover (being verified)

Someone will give you a **challenge code** (a UUID string). Do not generate your own.

**Step 1 — Sign the challenge**

Sign the exact challenge string as raw UTF-8 bytes using `SHA256withRSA` with your private key, then base64url-encode
the result (RFC 4648, no padding).

**Step 2 — Submit the proof**

```
POST https://api.tether.name/challenge/verify
Content-Type: application/json

{
  "challenge": "<the challenge code you were given>",
  "proof": "<your base64url-encoded signature>",
  "agentId": "<your agent ID>"
}
```

All three fields are required.

**Step 3 — Share the result**

If `valid` is `true`, share the `verifyUrl` with whoever challenged you:

```json
{
  "valid": true,
  "verifyUrl": "https://tether.name/check?challenge=<code>",
  "agentName": "YourName",
  "domain": "example.com",
  "registeredSince": 1700000000000
}
```

If `valid` is `false`, the `error` field tells you what went wrong:

| Error                            | Meaning                                                                                                                             |
|----------------------------------|-------------------------------------------------------------------------------------------------------------------------------------|
| `challenge_not_found_or_expired` | Challenge doesn't exist or expired (10 min TTL). Ask the challenger for a new one.                                                  |
| `challenge_already_used`         | This challenge was already verified. Challenges are single-use once successfully verified.                                          |
| `agent_not_found`                | The `agentId` doesn't match any registered agent. Check your agent ID.                                                              |
| `agent_missing_public_key`       | The agent exists but has no public key on file. Re-register with your owner.                                                        |
| `agent_owner_not_found`          | The agent's owner account was not found. Contact your owner.                                                                        |
| `signature_invalid`              | The signature didn't match. Make sure you're signing the exact challenge string with the correct private key using `SHA256withRSA`. The challenge is **not** consumed — you can retry with the same challenge code. |
| `internal_error`                 | Something went wrong on the server. Try again.                                                                                      |

### If you are the Challenger (verifying an agent)

**Step 1 — Generate a challenge**

```
POST https://api.tether.name/challenge
```

Response:

```json
{
  "code": "4e862e90-5053-42eb-8a83-2f856b2e85de"
}
```

**Step 2 — Send the challenge to the agent**

Give the agent the challenge code and ask them to verify. Example prompt:

> Please verify your identity on tether.name service. Sign this challenge and return the verification link.
> Challenge code: 4e862e90-5053-42eb-8a83-2f856b2e85de
> Verify endpoint: POST https://api.tether.name/challenge/verify
> Docs: https://tether.name/AGENTS.md

**Step 3 — Check the result**

The agent will return a verification link like `https://tether.name/check?challenge=<code>`. Click it or check
programmatically:

```
GET https://api.tether.name/challenge/<code>
```

Response:

```json
{
  "challenge": "4e862e90-5053-42eb-8a83-2f856b2e85de",
  "status": "verified",
  "agentName": "AgentName",
  "domain": "example.com",
  "registeredSince": 1700000000000,
  "verifiedAt": 1700000060000
}
```

Status will be `pending`, `verified`, `invalid`, or `not_found`.

## Signing Example (Shell/OpenSSL)

```bash
# Decode your base64 private key to a DER file
echo "$PRIVATE_KEY_BASE64" | base64 -d > /tmp/agent.der

# Convert PKCS8 DER to PEM (openssl dgst -sign requires PEM, not raw DER)
openssl pkcs8 -inform DER -in /tmp/agent.der -nocrypt 2>/dev/null \
  | openssl rsa -traditional -out /tmp/agent.pem 2>/dev/null

# Sign the challenge (use -w0 on Linux to disable base64 line wrapping)
PROOF=$(echo -n "$CHALLENGE" | openssl dgst -sha256 -sign /tmp/agent.pem \
  | base64 -w0 2>/dev/null || base64)
PROOF=$(echo -n "$PROOF" | tr '+/' '-_' | tr -d '=' | tr -d '\n')

# Clean up
rm /tmp/agent.der /tmp/agent.pem

# Submit
curl -X POST https://api.tether.name/challenge/verify \
  -H "Content-Type: application/json" \
  -d "{\"challenge\":\"$CHALLENGE\",\"proof\":\"$PROOF\",\"agentId\":\"$AGENT_ID\"}"
```

> **⚠️ Common pitfalls:**
> - `base64` on Linux wraps output at 76 characters by default, corrupting the proof. Use `base64 -w0` (Linux) or `base64` (macOS, which doesn't wrap by default).
> - `openssl dgst -sign` requires a PEM key file. Passing a PKCS8 DER file directly will silently produce an invalid signature. Always convert to PEM first.

## API Reference

| Endpoint                        | Method | Auth | Description                                     |
|---------------------------------|--------|------|-------------------------------------------------|
| `/challenge`                       | POST   | None | Generate a new challenge code (for challengers)      |
| `/challenge/verify`                | POST   | None | Submit a signed challenge (for provers)              |
| `/challenge/{code}`                | GET    | None | Check a challenge's verification status              |
| `/agents/{id}/register-key`        | POST   | None | Register your public key (requires registrationToken)|
| `/agents/{id}/status`              | GET    | Yes  | Check if your agent has a registered public key      |
| `/domains/claim`                   | POST   | Yes  | Claim a domain for DNS verification                  |
| `/domains/{id}/verify`             | POST   | Yes  | Trigger DNS lookup to verify domain ownership        |
| `/domains`                         | GET    | Yes  | List your claimed/verified domains                   |
| `/domains/{id}`                    | DELETE | Yes  | Remove a domain claim                                |
| `/AGENTS.md`                       | GET    | None | This document                                        |
| `/.well-known/tether-name.json`    | GET    | None | Machine-readable protocol spec                       |

## Request Format

All POST endpoints accept parameters via **JSON body**, **query parameters**, or **both**. If a field appears in both,
the JSON body value takes priority.

**JSON body (recommended):**
```
POST https://api.tether.name/challenge/verify
Content-Type: application/json

{"challenge": "...", "proof": "...", "agentId": "..."}
```

**Query parameters:**
```
POST https://api.tether.name/challenge/verify?challenge=...&proof=...&agentId=...
```

**Mixed (body fields override query params):**
```
POST https://api.tether.name/challenge/verify?agentId=abc123
Content-Type: application/json

{"challenge": "...", "proof": "..."}
```

## Checking Your Setup

After registering your public key, you can confirm your agent is ready by calling:

```
GET https://api.tether.name/agents/{agentId}/status
Authorization: Bearer <your API key>
```

Response:

```json
{
  "id": "your-agent-id",
  "agentName": "YourName",
  "registered": true
}
```

If `registered` is `false`, your public key wasn't stored — check that you used the correct registration token before it expired (5 minutes).

## Advertising Verification Support

To let others know you support Tether verification, include your agent ID in your system prompt, profile, or metadata:

```
tether: <your-agent-id>
```

For example, in a system prompt:

```
You are AgentName, a helpful assistant.
tether: sqBjJEo6T3RGqeh7MuPn
```

This lets challengers know they can ask you to verify your identity on tether.name without needing to guess.

For machine-readable discovery, you can also expose it in your agent's metadata or well-known files:

```json
{
  "tether": {
    "agentId": "sqBjJEo6T3RGqeh7MuPn",
    "verify": "https://tether.name/AGENTS.md"
  }
}
```

## Domain Verification (for Organizations)

If your owner has verified a domain via DNS, the verification response will show `domain` (e.g. `"domain": "example.com"`) instead of `email`. This provides organizational identity without exposing personal email addresses.

### How it works (for your owner)

1. **Claim a domain** at the [tether.name dashboard](https://tether.name) → Domains → "Verify a Domain"
2. **Add a DNS TXT record** — the dashboard provides the exact record:
   - **Host:** `_tether.example.com`
   - **Value:** `tether-verify=<token>`
3. **Verify** — click "Verify" once DNS propagates (usually a few minutes)

Once verified, all agents under that account will show the domain on verification results.

### API (for agents that manage domains programmatically)

| Endpoint                    | Method | Auth | Description                          |
|-----------------------------|--------|------|--------------------------------------|
| `/domains/claim`            | POST   | Yes  | Claim a domain, get DNS instructions |
| `/domains/{id}/verify`      | POST   | Yes  | Check DNS and verify ownership       |
| `/domains`                  | GET    | Yes  | List your claimed domains            |
| `/domains/{id}`             | DELETE | Yes  | Remove a domain claim                |

## Key Rotation & Compromise

If your private key is compromised or you need to rotate keys:

1. **Tell your owner immediately.** They can revoke your agent from the [tether.name dashboard](https://tether.name).
2. **Your owner issues a new agent** with a fresh registration token.
3. **Generate a new key pair** and register the new public key using the new registration token.
4. **Update your stored agent ID and private key** to the new values.
5. **Destroy the old private key.** Don't keep compromised keys around.

There is no self-service key rotation — your owner must be involved. This is by design: it ensures a human is always in the loop for identity changes.

## Key Rules

- **Challenges expire after 10 minutes.** Don't wait too long to sign.
- **Challenges are single-use once verified.** A successfully verified challenge cannot be reused. Failed attempts (e.g. `signature_invalid`) do **not** consume the challenge — you can retry.
- **Never generate your own challenge.** Only sign challenges given to you by others.
- **Keep your private key secret.** Never share it. If compromised, notify your owner immediately to revoke and reissue.
